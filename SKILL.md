# 高保真原型生成器

把用户的一句话系统描述，变成一个可运行、看起来像真实产品的高保真原型项目。

技术栈固定为：**Vue 3 + Vite + Element Plus + vue-router**。注意 Vue 3 只能用 Element Plus（Element UI 不支持 Vue 3），用户说"Element UI"时也用 Element Plus。

## 整体流程

### 1. 先问需求，再动手

用户要的是"高保真"，方向错了返工成本很高。动手前用提问工具一次问清 3~5 个关键点：

- 系统名称、用途、给谁用（角色）
- 包含哪些模块/页面（这会直接决定路由表）
- 核心操作流程（比如"从列表点进去看详情，然后审批"）
- 有没有必须出现的具体功能或字段

问题要少而关键，一次问完，不要来回追问。如果用户描述已经很具体（直接列了模块清单），跳过提问直接做。

### 2. 脚手架

运行脚本生成项目骨架（Vite + Vue3 + Element Plus + vue-router 已配好，含清新浅色主题和路由总览首页）：

```bash
bash <此skill目录>/scripts/create-project.sh <项目目录名> [父目录]
```

项目目录名用系统名的拼音或英文小写短横线形式，如 `order-admin`。父目录默认为当前工作目录。

脚本完成后项目里有：
- `src/router/index.js` — 路由表，**所有模块在这里注册**
- `src/views/Home.vue` — 首页，自动读取路由表生成原型卡片列表
- `src/App.vue` — 顶栏导航，自动读取路由表生成切换入口
- `src/styles/theme.css` — 清新浅色主题（CSS 变量 + Element Plus 主题色覆盖）

### 3. 逐个模块写页面

每个模块一个 `src/views/<ModuleName>.vue`，然后在 `src/router/index.js` 注册：

```js
{ path: '/orders', name: 'orders', component: () => import('../views/Orders.vue'),
  meta: { title: '订单管理', desc: '订单列表、详情与状态流转' } }
```

`meta.title` 和 `meta.desc` 必填——首页卡片和顶部导航都靠它们自动生成，路由注册完导航就通了，不用改别的文件。

**"高保真"的验收标准**（这是这个 skill 存在的意义，页面不能像线框图）：

- **Mock 数据要像真的**：张三李四王五、"杭州云杉网络科技有限公司"、¥12,800.00、真实的日期和状态流转，而不是"测试数据1""abc"。每页 10+ 条像样数据。
- **布局完整**：列表页 = 筛选区 + 表格 + 分页；详情用抽屉或对话框；表单带校验提示；有统计页就用卡片 + 图表。
- **交互要闭环**：点按钮要有反应（el-message 反馈、el-message-box 确认、对话框开合、步骤条流转）。没有后端，数据不用真的持久化，但页面内的交互路径必须走得通。
- **状态覆盖**：加载中（v-loading）、空数据（el-empty）至少关键页面要有。
- 优先用 Element Plus 现成组件（el-table、el-descriptions、el-tree、el-steps、el-tabs……），不要手写组件库里已有的东西。
- 需要图表时 `npm install echarts`，按需引入，配色沿用主题变量。

### 页面通用模式

生成列表/管理类页面时，统一遵守以下交互约定，保持原型体验一致：

**1. 筛选区放在页面顶部，统一收进一个 panel**

- 使用 `el-card` 或一个带白色背景、圆角、浅阴影的容器把整个筛选区包起来，放在表格上方。
- 内部用 `el-form` + `el-row`/`el-col` 实现 grid 布局：常规字段按 `xs:24 sm:12 md:8 lg:6` 自适应排列，日期范围、长搜索框等可占 `md:12 lg:8`。
- 筛选按钮（查询/重置）放在最后一列，与输入框同排或右对齐；查询用 `type="primary"`，重置用默认。
- 一个页面只保留这一个筛选 panel，不要把条件拆成多个区块。

**2. 点击表格行不跳转页面，改为弹 Dialog**

- 表格行加 `@row-click`，打开详情/编辑 Dialog，**不要**用 `router.push` 跳转到新页面（首页卡片和顶部导航已经是页面级切换）。
- Dialog 宽度按内容体量选择：
  - 小（30% / 420px）：纯文本详情、简单确认、少量字段展示。
  - 中（50% / 600px）：常规表单、编辑、含少量分组的详情。
  - 大（70% / 800px 或更大）：复杂详情、多 tab、时间线、商品/工单明细等需要横向空间的内容。
- 详情展示优先用 `el-descriptions`；编辑用 `el-form`。Dialog 标题写明当前对象名，如 "订单详情：ORD-20240719-001"。

**3. 删除必须二次确认**

- 任何删除按钮（行内删除、批量删除、卡片删除）都必须先调用 `ElMessageBox.confirm`，文案明确告诉用户"删除后不可恢复"及即将删除的对象名称。
- 确认后再执行本地删除（更新 tableData 数组或清空当前项）并给出 `ElMessage.success` 反馈；取消时不做任何操作。
- 危险操作按钮统一用 `type="danger"`，行内删除可用文字按钮 `link type="danger"`。

### 4. 风格

遵循 `references/style-guide.md`（清新浅色：浅色背景、圆角卡片、充足留白、青绿色主色）。视觉定制集中改 `src/styles/theme.css` 的 CSS 变量，页面里不要到处写死颜色。

**顶部导航不要冒出多余滚动条**：`App.vue` 的 `.nav` 使用 `overflow-x: auto` 时，**必须同时写 `overflow-y: hidden`**。CSS 规范里只要 `overflow-x`/`overflow-y` 其中之一不是 `visible`，另一个就会被浏览器计算成 `auto`；而 `.nav-item` 带的 `border-bottom` 会让内容比容器高 1px，结果顶部导航会冒出一条多余的垂直滚动条。模板已默认带 `overflow-y: hidden`，**改 `App.vue` 时不要删掉这一行**。同类问题：任何"只想要横向滚动"的容器（标签栏、横向步骤条、移动端 tab 条）都要 `overflow-x: auto; overflow-y: hidden;` 成对写。

### 5. 验证能跑

交付前必须验证，不能交付一个跑不起来的"原型"：

1. `npm run build` 必须通过，有报错修到通过为止
2. `npm run dev` 后台启动，确认 dev server 正常监听
3. 把访问地址（如 http://localhost:5173）和项目路径交付给用户，简要说明有哪些页面

## 注意

- 每次"生成高保真原型"都是**独立新项目**，不要往已有项目里塞。
- 原型是给产品/业务看的，美观和真实感的优先级高于代码优雅；但也不要写一打开控制台全是 warning 的代码。
- 如果用户中途改需求（加模块、改风格），直接在当前项目上增量改，不用推倒重来。
