# 高保真原型生成器

把用户的一句话系统描述，变成一个可运行、看起来像真实产品的高保真原型项目。

技术栈固定为：**Vue 3 + Vite + Element Plus + vue-router**。注意 Vue 3 只能用 Element Plus（Element UI 不支持 Vue 3），用户说"Element UI"时也用 Element Plus。

## 设计思路构思
正式编码前，先梳理项目背景，并确立一套**极具辨识度**的视觉风格方向：
1. **使用目的**：该界面需要解决什么问题？目标使用者是谁？
2. **视觉调性**：选定一种极致鲜明的风格：极简克制、繁复饱和、复古未来风、自然有机风、轻奢精致风、童趣玩具风、杂志版式风、粗野原始风、装饰艺术几何风、柔和马卡龙风、工业实用风等。以上仅作灵感参考，最终设计需贴合你选定的核心风格落地。
3. **开发约束**：技术硬性要求（开发框架、性能指标、无障碍访问规范）
4. **差异化亮点**：设计中让人过目不忘的核心特色是什么？用户看完唯一能记住的设计亮点是什么？

**核心要求**：选定清晰统一的设计概念，并完整、精细地落地执行。浓烈繁复的饱和风、克制干净的极简风都能出彩，关键在于每一处设计都有明确意图，而非单纯堆砌视觉效果。

随后输出可运行代码（HTML/CSS/原生JS、React、Vue 等），代码需满足以下标准：
- 符合线上生产环境标准，功能完整可用
- 视觉冲击力强，具备独特记忆点
- 整套界面视觉统一，拥有完整、鲜明的设计立场
- 所有视觉细节经过精细打磨

## 前端视觉设计规范
重点打磨以下维度：
### 字体排版
选用美观、独特、富有设计感的字体。避免 Arial、Inter 这类烂大街的通用字体；优先选择风格鲜明、自带辨识度的字体，提升整体界面质感；搭配一套个性标题字体 + 一套精致正文字体形成层级对比。

### 色彩与主题
整套界面统一视觉调性，使用 CSS 变量统一管理色值。主色调搭配高对比点缀色，效果远优于平淡、均分饱和度的平庸配色方案。

### 动效设计
通过动画实现页面特效与微交互；纯 HTML 页面优先使用 CSS 实现动效；React 项目可搭配 Motion 动画库。重点打造高记忆点动画：例如一套分层延迟出场的页面加载动画，远比零散细碎的小交互更能带来高级体验；可添加滚动触发动画、超出预期的悬浮反馈效果。

### 空间构图
打破常规布局，尝试不对称构图、图层叠加、斜向视觉动线、突破网格约束的元素；要么预留大量留白营造松弛感，要么高密度紧凑排布营造氛围感，二选一贯彻到底。

### 背景与视觉装饰
打造画面层次与氛围感，不要直接使用纯色背景。搭配贴合整体风格的氛围特效与纹理，例如渐变网格、杂色颗粒、几何纹样、多层透明叠加、高对比度阴影、定制装饰边框、自定义鼠标光标、胶片颗粒遮罩等创意视觉元素。

## 绝对禁止的流水线AI设计套路
切勿使用烂大街的AI通用设计模板：
1. 泛滥通用字体（Inter、Roboto、Arial、系统默认字体）
2. 千篇一律配色（尤其白底搭配紫调渐变）
3.  predictable layouts 毫无新意的常规布局、组件模板
4. 无专属场景特色的流水线式标准化设计

## 创意落地要求
灵活发挥创意，做出出人意料、贴合项目场景的设计方案。每一套界面都要有专属设计，不可同质化；灵活切换亮/暗黑主题、更换字体、变换视觉风格。**禁止**多次输出时反复复用同款热门设计（例如 Space Grotesk 字体）。

**重要提示**：代码复杂度需匹配设计视觉定位。繁复饱和风格需要大量代码实现丰富动画与视觉特效；极简精致风格则需要克制的代码，精准把控间距、文字排版与细微视觉细节。高级质感来源于完整贯彻设计理念、精细落地每一处细节。

请记住：本工具可实现极具创意的优质设计，不要局限思路，跳出固有模板思维，完整发挥独特设计能力。

## 整体流程

### 1. 先问需求，再动手

用户要的是"高保真"，方向错了返工成本很高。动手前用提问工具一次问清 3~5 个关键点：

- 系统名称、用途、给谁用（角色）
- 包含哪些模块/页面（这会直接决定路由表）
- 核心操作流程（比如"从列表点进去看详情，然后审批"）
- 有没有必须出现的具体功能或字段

问题要少而关键，一次问完，不要来回追问。如果用户描述已经很具体（直接列了模块清单），跳过提问直接做。

### 2. 脚手架

运行脚本生成项目骨架（Vite + Vue3 + Element Plus + vue-router 已配好，含清新浅色主题和路由总览首页）。按平台二选一：

```bash
# macOS / Linux（bash）
bash <此skill目录>/scripts/create-project.sh <项目目录名> [父目录]
```
```powershell
# Windows（PowerShell）
pwsh <此skill目录>/scripts/create-project.ps1 <项目目录名> [父目录]
# 或在 PowerShell 中直接: ./scripts/create-project.ps1 <项目目录名> [父目录]
```

项目目录名用系统名的拼音或英文小写短横线形式，如 `order-admin`。父目录默认为当前工作目录。

两个脚本开头都会先自检 Node 环境：若本机没有 Node 或大版本不是 22，会自动安装 **v22.23.x** 到用户目录（免 sudo / 免管理员权限），并把激活配置写进 shell profile，保证后续新 shell 也能用 `node`/`npm`；已有合用的 Node 则直接复用，不动用户环境。差异只在版本管理器：mac/Linux 用 nvm，Windows 用 fnm。需要 pnpm 时手动跑一次自检脚本加 `--pnpm`（默认流程走 npm）。

脚本完成后项目里有：
- `src/router/index.js` — 路由表，**所有模块在这里注册**
- `src/views/Home.vue` — 首页，自动读取路由表生成原型卡片列表
- `src/App.vue` — 顶栏导航，自动读取路由表生成切换入口
- `src/styles/theme.css` — 清新浅色主题（CSS 变量 + Element Plus 主题色覆盖）

**Element Plus 已按需引入**（`vite.config.js` 配了 `unplugin-vue-components` + `ElementPlusResolver`），不要写 `app.use(ElementPlus)` 或全量 `import 'element-plus/dist/index.css'`。约定：
- 模板里用到的 `el-*` 组件（含 `v-loading` 指令）**自动注册 + 自动引入各自样式**，直接写标签即可，无需 import、无需局部注册。
- 中文 locale 由 `App.vue` 的 `<el-config-provider :locale="zhCn">` 承接，不要再改回 `app.use` 方式。
- `ElMessage` / `ElMessageBox` / `ElNotification` 这类 JS API 仍需在用到的地方显式 `import { ElMessage } from 'element-plus'`（其样式已在 `main.js` 统一引入）。

### 3. 逐个模块写页面

每个模块一个 `src/views/<ModuleName>.vue`，然后在 `src/router/index.js` 注册：

```js
{ path: '/orders', name: 'orders', component: () => import('../views/Orders.vue'),
  meta: { title: '订单管理', desc: '订单列表、详情与状态流转' } }
```

`meta.title` 和 `meta.desc` 必填——首页卡片和顶部导航都靠它们自动生成，路由注册完导航就通了，不用改别的文件。

**"高保真"的验收标准**（这是这个 skill 存在的意义，页面不能像线框图）：

- **Mock 数据要像真的**：张三李四王五、"杭州云杉网络科技有限公司"、¥12,800.00、真实的日期和状态流转，而不是"测试数据1""abc"。每页 10+ 条像样数据。
- **布局完整**：列表页 = 筛选区 + 表格 + 分页；详情用抽屉或对话框；表单带校验提示；有统计页就用卡片 + 图表。
- **交互要闭环**：点按钮要有反应（el-message 反馈、el-message-box 确认、对话框开合、步骤条流转）。没有后端，数据不用真的持久化，但页面内的交互路径必须走得通。
- **状态覆盖**：加载中（v-loading）、空数据（el-empty）至少关键页面要有。
- 优先用 Element Plus 现成组件（el-table、el-descriptions、el-tree、el-steps、el-tabs……），不要手写组件库里已有的东西。
- 需要图表时 `npm install echarts`，按需引入，配色沿用主题变量。

### 页面通用模式

生成列表/管理类页面时，统一遵守以下交互约定，保持原型体验一致：

**1. 筛选区放在页面顶部，统一收进一个 panel**

- 使用 `el-card` 或一个带白色背景、圆角、浅阴影的容器把整个筛选区包起来，放在表格上方。
- 内部用 `el-form` + `el-row`/`el-col` 实现 grid 布局：常规字段按 `xs:24 sm:12 md:8 lg:6` 自适应排列，日期范围、长搜索框等可占 `md:12 lg:8`。
- 筛选按钮（查询/重置）放在最后一列，与输入框同排或右对齐；查询用 `type="primary"`，重置用默认。
- 一个页面只保留这一个筛选 panel，不要把条件拆成多个区块。

**2. 点击表格行不跳转页面，改为弹 Dialog**

- 表格行加 `@row-click`，打开详情/编辑 Dialog，**不要**用 `router.push` 跳转到新页面（首页卡片和顶部导航已经是页面级切换）。
- Dialog 宽度按内容体量选择：
  - 小（30% / 420px）：纯文本详情、简单确认、少量字段展示。
  - 中（50% / 600px）：常规表单、编辑、含少量分组的详情。
  - 大（70% / 800px 或更大）：复杂详情、多 tab、时间线、商品/工单明细等需要横向空间的内容。
- 详情展示优先用 `el-descriptions`；编辑用 `el-form`。Dialog 标题写明当前对象名，如 "订单详情：ORD-20240719-001"。

**3. 删除必须二次确认**

- 任何删除按钮（行内删除、批量删除、卡片删除）都必须先调用 `ElMessageBox.confirm`，文案明确告诉用户"删除后不可恢复"及即将删除的对象名称。
- 确认后再执行本地删除（更新 tableData 数组或清空当前项）并给出 `ElMessage.success` 反馈；取消时不做任何操作。
- 危险操作按钮统一用 `type="danger"`，行内删除可用文字按钮 `link type="danger"`。

### 4. 风格

遵循 `references/style-guide.md`（清新浅色：浅色背景、圆角卡片、充足留白、青绿色主色）。视觉定制集中改 `src/styles/theme.css` 的 CSS 变量，页面里不要到处写死颜色。

**顶部导航不要冒出多余滚动条**：`App.vue` 的 `.nav` 使用 `overflow-x: auto` 时，**必须同时写 `overflow-y: hidden`**。CSS 规范里只要 `overflow-x`/`overflow-y` 其中之一不是 `visible`，另一个就会被浏览器计算成 `auto`；而 `.nav-item` 带的 `border-bottom` 会让内容比容器高 1px，结果顶部导航会冒出一条多余的垂直滚动条。模板已默认带 `overflow-y: hidden`，**改 `App.vue` 时不要删掉这一行**。同类问题：任何"只想要横向滚动"的容器（标签栏、横向步骤条、移动端 tab 条）都要 `overflow-x: auto; overflow-y: hidden;` 成对写。

### 5. 验证能跑

交付前必须验证，不能交付一个跑不起来的"原型"：

1. `npm run build` 必须通过，有报错修到通过为止
2. `npm run dev` 后台启动，确认 dev server 正常监听
3. 把访问地址（如 http://localhost:5173）和项目路径交付给用户，简要说明有哪些页面

## 设计原则

源自 Arco Design 设计原则，落地到原型生成的具体动作。写每个页面时逐条对照一遍，不达标就改到达标（与上面的"创意视觉"不冲突——再独特的设计也要保证可用性）：

1. **及时反馈**——任何按钮点击、表单提交、状态变更都要有即时反馈：`el-message` 提示、按钮 `:loading` 态、`el-message-box` 确认、开关/步骤条的即时联动。不能出现"点了没反应"的死按钮。
2. **贴近现实**——用业务真实用语和真实数据，不用"测试1""字段A""abc"。日期写"2024-07-19 14:30"，金额写"¥12,800.00"，公司写"杭州云杉网络科技有限公司"，状态写"待审批/已通过/已拒绝"而不是"状态1/状态2"。
3. **系统一致性**——同一动作全站只用一个词（用"删除"就别混用"移除""清除"）；同类信息呈现方式一致（所有列表页的筛选 panel 长一个样）；同一状态用同一种 el-tag 颜色（待审批=warning、已通过=success、已拒绝=danger、进行中=primary）。注意：一致性指"同一个原型内部统一"，不限制跨项目用不同视觉风格。
4. **防止错误发生**——比报错更好的是别让用户出错：不可用按钮 `disabled` 并带 `el-tooltip` 说明原因；危险操作（删除/审批/发布/上下架）二次确认；表单提交前先前端校验；金额、数量、手机号等关键字段限制输入类型与范围；不可逆操作按钮用 `type="danger"`。
5. **遵从习惯**——动作和选项要可见、别让用户记忆：选中行后批量操作按钮出现在表格上方工具栏，不藏在右键菜单里；填完表单点"提交"就是提交，不要多塞一层"确认提交"的中间步骤；分步流程用 `el-steps` 让用户一眼看到当前在第几步。
6. **突出重点**——用户是"扫"不是"读"。页面一打开能看到核心信息（统计数字、待办、异常告警放在顶部）；次要信息折叠或弱化（次要文字用 `#94a3b8`、辅助说明小字 12px）；一屏不超过 3 个视觉重点；按钮区只突出一个 primary 主操作，其余用 default。
7. **错误帮助**——校验失败用"请输入正确的手机号，如 13800138000"而不是"格式错误"；空数据用 `el-empty` 配可操作按钮（"暂无订单，去新建"）；加载失败给"重试"按钮；表单字段错误提示紧贴字段并说清怎么改。
8. **人性化帮助**——能用 `placeholder`、`el-tooltip` 说清楚的别写帮助文档；首次进入用 `el-alert` 或引导卡片提示关键动作；复杂字段配问号图标 tooltip；帮助层级遵循：无需提示 > 一次性提示 > 常驻提示 > 帮助文档。

## 文案与用语规范

原型里所有文字（按钮、表头、提示、mock 数据）都要像真实产品文案，不是占位符。基于 Arco 用语原则，无论选用哪种视觉风格都适用：

- **操作文案 = 动词 + 名词**："新建订单""导出报表""审批通过""批量删除""保存草稿"，不要写成名词短语或光秃秃一个动词。
- **统计文案 = 数字 + 单位 + 动词/名词**："待处理订单 12 单""今日成交 ¥48,200""逾期工单 3 个""本月新增客户 28 家"。
- **不用敬称「您」**：用"你"或省略主语。写"请输入手机号"而不是"请您输入手机号"。
- **默认状态用正向表述**：规则提示用正向（"金额需大于 0""最多可选 5 项"），不用"金额不能为空""最多不能超过 5 项"；只有真正禁止时才用"请勿"。
- **日期时间**：统一 24 小时制，格式 `YYYY-MM-DD HH:mm`；日期范围用 `~` 连接，如"2024-07-01 ~ 2024-07-31"；时间戳加"刚刚 / 5 分钟前 / 今天 14:30"等相对表述更真实。
- **数字**：四位以上用千分撇分节（"共 12,800 条"）；金额带货币符号且保留两位小数（"¥12,800.00"）；百分比保留一致小数位（"85.6%"）；ID 类用业务化前缀（"ORD-20240719-001""WO-2407-0042"）。
- **中英文混排加空格**："导出 Excel 报表""共 12 单""提交 PDF 附件"；但"3D 打印""7 号线""IPv4"这种不表数量的不加空格。
- **词汇统一**：同一事物全站只用一个词（"商品"就别有时叫"产品"有时叫"货品"）；入口文案和页面标题要相关一致，不能毫无关联；同一状态在列表、详情、统计里叫法一致。

## 注意

- 每次"生成高保真原型"都是**独立新项目**，不要往已有项目里塞。
- 原型是给产品/业务看的，美观和真实感的优先级高于代码优雅；但也不要写一打开控制台全是 warning 的代码。
- 如果用户中途改需求（加模块、改风格），直接在当前项目上增量改，不用推倒重来。

## 系统管理类模块补充

当用户要求生成“系统管理 / 后台管理 / 权限管理 / 基础管理”类原型，或需求中包含用户、角色、菜单、部门、字典、日志、定时任务等常见系统管理模块时，在写页面前先读取 `references/admin-system-patterns.md`。该参考文件定义了这些模块的通用结构、字段、交互约定和 Mock 数据规则，确保生成的系统管理类原型既真实又一致。

读取该参考后，按常规流程生成页面：每个模块一个 `src/views/<ModuleName>.vue`，路由注册到 `src/router/index.js`，并保持整体清新浅色风格。只生成用户明确提到的模块，不要画蛇添足添加无关的系统管理页面。
